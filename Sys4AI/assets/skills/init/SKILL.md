# Generic /init

## Purpose

A target system generated by Sys4AI may adapt `/init` as the user-facing front door for system definition and adoption.

The skill classifies the situation, gathers evidence, identifies candidate strategic-intent and approval gaps, summarizes the route, and asks for approval before any controlled artifact or scaffold is created.

This template is not itself active development-system authority unless promoted through the local system's skill registry and runtime policy.

## When to use

Use this skill when:

- the user invokes `/init`, `/init greenfield`, `/init brownfield`, or `/init temp_prd`,
- a new system needs discovery before downstream requirements or implementation work,
- an existing system or repository needs read-only classification before Sys4AI governance adoption,
- the system-of-interest, subject layer, lifecycle goal, evidence base, or downstream route is unclear.

Do not use this skill to generate final Product Requirements Documents, system requirements, implementation plans, scaffolds without explicit approval.

## Inputs

- User prompt or `/init` invocation.
- Optional repository or document evidence.
- Optional prior `temp_prd.md`, Requirements Discovery Record, Current-State Baseline, or handoff.
- Existing project authority hierarchy, if available.

## Outputs

- Situation classification: `greenfield`, `brownfield`, `partially_built`, or `documentation_recovery`.
- System-of-interest and subject layer.
- Lifecycle intent: `build`, `improve`, `maintain`, `operate`, `migrate`, or `recover`.
- Candidate strategic-intent posture for new or substantially changed targets.
- Evidence summary.
- Approval prompt before writing controlled discovery or downstream artifacts.
- Downstream route recommendation.

## Procedure

1. Classify the situation and lifecycle goal.
2. Identify the system-of-interest.
3. Classify the subject layer.
4. Identify mission-versus-vision, `VISION-CAND-*` and `VALUE-CAND-*` needs, anti-values, source and inference state, missing stakeholders, accountable approval principal, inherited constraints, conflicts, waiver state, and review cadence. Do not imply approval.
5. Inspect available source evidence before asking questions.
6. For greenfield systems, ask what system the user wants to develop when missing.
7. For brownfield systems, perform the first pass in read-only inspection mode.
8. Summarize classification and evidence in chat before writing any controlled artifact.
9. Preserve candidate requirements as `REQ-CAND-*` or `NFR-CAND-*`.
10. Ask before writing a Requirements Discovery Record:

```text
I have enough evidence to create a draft Requirements Discovery Record. Should I write it to the controlled discovery area? This will not modify source code or install scaffolding.
```

11. Ask before creating a Product Requirements Document:

```text
Discovery is complete. Should I create a Product Requirements Document with `/conversation-to-prd` using the current discovery evidence?
```

12. Ask before brownfield governance adoption planning:

```text
For this brownfield project, should I create an implementation plan for Sys4AI governance adoption?
```

13. Route to downstream skills only after approval.

## Brownfield first-pass rule

The first brownfield pass is read-only. Inspect repository structure, README, Product Requirements Documents, implementation plans, tests, Makefile or CLI entry points, schemas, registries, docs, architecture hints, risks, and existing governance surfaces. Do not write files until the user or Director approves a controlled artifact.

## Authority boundaries

- `/init` is a routing and discovery gate, not a scaffold generator.
- Current-State Baselines and Requirements Discovery Records are draft evidence unless promoted.
- Candidate requirements are not baselined requirements.
- Generated derivatives do not authorize source changes.
- Silence, model authorship, controlled-file location, and structural validation do not approve candidate vision or values.
- Product Requirements Document synthesis, system requirements, implementation plans, scaffolds require explicit approval.

## Validation

Run:

```bash
cd Sys4AI
make validate-assets
```

If a discovery record is produced, also run the discovery-record validator when available.
