# Docker Decision Guide

**Status:** Draft  
**Scope:** Development environment and generated target-system runtime templates

---

## Recommendation for Phase 1

Use `.venv` first. Defer Docker.

The current project needs Python scripts, YAML records, skill adapters, memory registries, validators, and documentation policies. That is a small Python workshop, not yet a container fleet.

---

## When to add a development container

Add `.devcontainer/` or `docker/dev/Dockerfile.dev` when the development workflow needs:

- OS packages.
- LaTeX or PDF build tooling.
- Java for PlantUML.
- Graphviz.
- Browser automation.
- A database or vector index.
- Multi-service orchestration.
- Strict CI/local parity.

---

## When to add target-runtime templates

Add or expand `templates/target_runtime/` when `sys-for-ai` begins generating deployable target agentic systems that need containers.

Keep target-runtime templates separate from the development environment. A development container builds the framework. A target-runtime container runs a system generated by the framework.

---

## Phase 1 decision

```text
Baseline: .venv
Docker: deferred
Target runtime templates: placeholder only
Revisit: Phase 2 or when a Docker trigger appears
```
